## Summary

This stopping point corrects the Theorem 3.3 proof path. The earlier one-scale
materialized-supply wrappers are useful staging lemmas, but they are stronger
than the paper: they require the rounded supply total to be `#agents * L`,
where the same `L` is also used for the value grid. The source proof instead
rounds values using the original average and then applies Claim 3.4 to the
rounded instance's actual average.

The faithful route now has explicit support:

- `L0`: original average/value-grid scale.
- `LR`: rounded-instance average/window scale.
- a finite capped two-scale `U`/`V(U)`/value-pair search API.
- a Claim 3.4 bridge that runs on materialized rounded supply values computed
  from `L0` but bounded around `LR`.

No post-validation dashboard/report pass was run; the paper is not done.

## Source Fact To Preserve

From the source text around Theorem 3.3:

- Original average `L` is used to define rounded values `kL/lambda^2`.
- The rounded instance defines its own average `LR`.
- The paper says `L <= LR`.
- Claim 3.4 is applied to the rounded instance average `LR`.
- The finite type set `U` uses the window `LR/2 < v(t) < 2LR`.

The old exact-total route with `#agents * L0` is a staging abstraction, not the
faithful source path.

## Next Proof Path

1. Define a paper-facing rounded-average wrapper for a supply:
   `weightedRoundedSupply L0 lambda supply = #agents * LR`.
   You can keep `LR` as an explicit parameter with the equality premise; no
   need to divide by `Fintype.card` unless a later theorem needs the definition.

2. Use
   `paper_lmms_claim_3_4_exists_bounded_optimal_for_rounded_supply_average_no_top`
   with:
   - `hsupply_average` at `LR`,
   - `hbase_le_avg : L0 <= LR`,
   - no-top support,
   - `0 < L0`, `0 < LR`, `0 < lambda`.

3. From the returned bounded allocation `B`, form the min/max pair using
   `roundedBundleTypeLoad L0`, but prove pair membership in
   `roundedAdmissibleValuePairSetWithCap L0 LR lambda maxCount`, not the old
   one-scale `roundedAdmissibleValuePairSet L0 lambda`.

4. Prove the cap assumption
   `2 * LR <= (maxCount + 1) * (L0 / lambda)` from the source rounding error
   bounds. A conservative cap can depend on `lambda` and the known inequality
   `LR <= L0 + highError + lowError` if that is easier than optimizing the
   count bound. The source only needs "constant in lambda" for the IP dimension.

5. Build a two-scale concrete IP certificate from the exact bounded allocation
   and run
   `exists_roundedValuePairSearchCertificateWithCap_of_feasible_pair`.

6. Only after the two-scale feasible-pair search bridge compiles, return to the
   combined high/low supply and prove the source facts:
   - `L0 <= LR`,
   - strict item smallness for Claim 3.4 via no-top support plus `L0 <= LR`,
   - the chosen cap covers `2LR`,
   - coordinate counts match the combined supply.

7. Leave Lenstra/runtime and final PTAS/FPTAS complexity certificates for the
   next layer. Do not conflate the finite search certificate with the runtime
   proof.
