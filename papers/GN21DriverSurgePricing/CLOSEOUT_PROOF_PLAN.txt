Meaning of r1_current
---------------------
r1_current is the measured one-state reward rate of the current fixed
non-surge policy:

  gn21MeasuredStateRewardRate
    (μ 0) (arrival 0)
    (ctmcStructuredDynamicSurgePrice m z switch12 switch21 0)
    (ρ 0)

It is not a raw expected payment.  It is W_1(σ_1) / T_1(σ_1), where W_1 is
average payment per accepted trip and T_1 is the state cycle time.

The paper's Lemma 9 fixes arbitrary σ_1, and thus local Q_1, T_1, R_1
(`source.txt:3809`).  The Theorem 3 statement separately uses target earning
rates R_1 < R_2 (`source.txt:3944`).  In formal names and scratch work, call
the Lemma 9 local quantity `r1_current` or `R1_current`, and reserve `R1` /
`targetR1` for the Theorem 3 target rate.  The local fixed-state reward rate
is not the target accept-all R1 unless σ_1 is already accept-all or a separate
equality has been proved.

Important conclusion from the source proof
------------------------------------------
This is not a claim that the paper proof is wrong.  The paper's Theorem 3
proof uses compact notation: Lemma 9 fixes an arbitrary current non-surge
policy, while the displayed Theorem 3 quantities are accept-all primitives.
Lean therefore needs the missing bridge to be explicit, either by proving the
current-policy quantities agree with the required fixed-transfer data on the
relevant branch, or by routing through Theorem 4's shape/endpoint reduction.

Do not try to close Lemma 9 by proving a universal arbitrary-policy bound

  r1_current <= R1.

That is not the right invariant, and for arbitrary current non-surge policies
it can be false.  The usable paper argument is the Theorem 3 surge-side
"support any ratio" paragraph: choose the surge-state multiplier/intercept with
enough slack so that

  z_2 / (m_2 - r1_current)

stays inside the current Lemma 9 interval whenever the current non-surge reward
is below a verified envelope Rmax.  In Lean, the target is therefore:

  r1_current <= Rmax
  z_2 < current_Lemma9_upper * (m_2 - Rmax)

plus current lower-endpoint nonpositivity.  The compiled constructor for this
route is:

  GN21SurgeLemma9AcceptAllAggregateRewardRateData.exists_of_reward_envelope_current_lower_upper_slack

Step 2: prove pointwise nonnegativity of the constructed non-surge price.
Done.

The difficult-looking issue is that Theorem 3 allows z_1 to be negative.  This
is still manageable from the Lemma 10 lower bound.

For the non-surge price,

  w_1(τ) = R2 * τ + z_1 * q_{1->2}(τ)
         = R2 * (τ + ratio * q_{1->2}(τ))

where z_1 = ratio * R2 and R2 > 0.

If ratio >= 0, use q >= 0 and τ > 0.

If ratio < 0, use q_{1->2}(τ) <= switch12 * τ.  It is enough to show

  1 + ratio * switch12 >= 0.

The Lemma 10 lower bound gives

  ratio > -(T2*switch12 + Q2)
            / (Q2*(switch12*T1 - Q1) + switch12*(T2*switch12 + Q2)).

Because Q2 > 0 and switch12*T1 - Q1 > 0 for the accept-all non-surge
primitive, the denominator is strictly larger than

  switch12 * (T2*switch12 + Q2).

Therefore the lower bound is strictly greater than -1/switch12.  Hence

  ratio > -1/switch12
  1 + ratio*switch12 > 0

and pointwise w_1(τ) >= 0 follows for all positive τ.
